# Editorial Guide v0

このドキュメントは、本 DB(ビジネス史・産業史学習データベース)に事例を入力するときの統一ルール。自分が後日読み返しても意味を取り違えないための「執筆OS」。最初の3件を書きながら更新していく前提の生きたドキュメント。

## 0. 基本原則

1. **事実と解釈を分離する**。`Event.description` は事実、本文 `## 考察・私見` は解釈。
2. **構造を先に、物語を後に**。物語に流されると比較可能性が死ぬ。
3. **あとで迷うくらいなら書かない**。draft で寝かせてよい。
4. **出典のない記述は暫定**。published にする時までに少なくとも1つは紐付ける。
5. **DB と Markdown の分離**を守る: 本文は Markdown、構造データは DB が正本。

## 1. タイトルの付け方

- **形式**: 「{主題}の成立史」または「{主題}と{変化の種類}」
- **悪い例**: 「TDL について」「コンテナの話」
- **良い例**: 「東京ディズニーランドの成立史」「コンテナ化と世界海運の構造転換」
- **長さ**: 20〜30文字を目安、40文字を超えない
- **固有名詞は正式名称**: 「TDL」ではなく「東京ディズニーランド」(通称はタグにする)

## 2. 一言要約(one_liner)の書き方

- **60字以内**(できれば50字以内)
- **形式**: 「{主題}が、{条件}で、{構造転換}を起こした事例」のように「転換の型」を入れる
- **悪い例**: 「TDL は大成功したテーマパーク」(何が起きたか不明)
- **良い例**: 「浦安の埋立地と米国IPを出資なしのライセンス契約で結び、アセット保有と運営を分離した日本型テーマパーク」

## 3. Event の粒度

- **1年あたり1〜3件が目安**。Wikipedia の年表並みに細かくしない。
- **「誰が何を決めた / 何が起きた / 何が変わった」の最小単位**で1 Event。
- **粒度を上げる基準**: その Event がなかったら後の展開が違っていたか?
- **粒度を下げる基準**: その Event は別の Event の詳細にすぎないか?
- **重複 Event の禁止**: 同じ出来事を複数事例に書きたい時は Entity か CaseRelation で表現する。
- **1事例あたり5〜12件を目安**。それ以上は本当は2つの事例かもしれない。

## 4. Turning Point(転換点)の判断基準

`is_turning_point: true` にしてよいのは、次の3つのどれかを満たす時:

1. **レジーム変化**: 事例の主要な力学(誰が勝つか、何が価値を生むか)が変わった
2. **不可逆性**: その後の展開が、この Event なしには成立しない
3. **外部要因による強制**: 規制・事故・政変など、自発的でない変化

**1事例あたり最低1、最大4** の転換点を目安に。4 を超える場合は粒度を見直す(本当は違う事例が混ざっているかもしれない)。

> **比較ビューの都合**: 転換点の `description` フィールドは**必ず書く**。比較ページでは year + title + description が横並びで表示されるため、description が空だとその行の情報量が激減する。1〜3文で「なぜそれが起きたか」を事実ベースで書けばよい。

## 5. Theme と Tag の使い分け

| | Theme | Tag |
|---|---|---|
| 粒度 | 大(10〜20個) | 小(無制限) |
| 増やし方 | 編集者が意図的に | 書きながら自由に |
| ナビ | ホーム画面の主要導線 | フィルタ・検索の副次導線 |
| 階層 | あり(parent) | なし(ただしカテゴリあり) |
| 例 | 「アセットとオペレーションの分離」 | `pattern/standardization`, `industry/shipping` |

**判断基準**: ホーム画面の Theme カードに並べたいか? → Theme。検索で絞り込みに使いたいか? → Tag。両方 → まず Tag、多くの事例に付くようになったら Theme に昇格。

### Tag のカテゴリ規約

`tag.slug` は `category/name` 形式を推奨:
- `pattern/xxx` — 構造パターン(standardization, vertical-integration 等)
- `industry/xxx` — 業界(shipping, railway, theme-park 等)
- `region/xxx` — 地理(japan, global, east-asia 等)
- `era/xxx` — 時代(postwar-showa, heisei, edo 等)
- `mechanism/xxx` — 背後の力学(network-effect, regulatory-capture 等)

### Tag プレフィックス: `driver/` と `scope/`（比較分類用）

比較ビューの「変化の型」行に表示される2軸。各事例に1〜2個ずつ付ける。

- `driver/personal-obsession` — 創業者・トップ個人の執念が起動因
- `driver/disruptive-entry` — 既存業界外の参入者が破壊的に参入
- `driver/cost-pressure` — コスト構造の劣化・競争劣位が圧力になった
- `driver/technology-shift` — 技術の変化が引き金
- `driver/latecomer-advantage` — 後発であること自体が有利に働いた
- `scope/mega-project` — 一つの巨大プロジェクトの成否
- `scope/industry-restructuring` — 産業全体の構造が変わった
- `scope/platform-ecosystem` — プラットフォームとエコシステムの形成・崩壊

## 6. Lesson の書き方

- **形式**: 「{条件}のとき、{結果}になりやすい」または「{主張}」
- **1文で言い切る**(補足は `applies_to` に)
- **LessonKind の選び分け**:
  - `PATTERN`: 「こうなりがち」の型
  - `ANTI_PATTERN`: 「これをやると失敗しがち」の型
  - `MECHANISM`: 「なぜそうなるか」の力学
  - `FORECAST_METHOD`: 「未来を予測するときに使えるレンズ」
  - `HEURISTIC`: 「判断に使える経験則」
- **禁止**: 「〜が重要である」で終わる Lesson(情報量ゼロ)
- **禁止**: 事例固有の感想(「TDL は大成功した」) — これは Lesson ではない
- **推奨**: 反証可能な書き方、他の事例に適用できる抽象度
- **テスト**: この Lesson を別の事例(例: SaaS 黎明期)に当てはめて意味が通るか? 通るなら良い Lesson。

## 6b. Entity の役割(role)の使い分け

比較ビューの「プレイヤー構造」行に表示される3役割が特に重要:

| role | 表示 | 意味 | 例 |
|---|---|---|---|
| `PROTAGONIST` | 主役（緑） | 変化を推進・成功させた側 | OLC、十河信二、McLean |
| `ANTAGONIST` | 拮抗（赤） | 変化に抵抗したが敗れた側 | 大蔵省、港湾労組、全海員 |
| `DISRUPTOR` | 破壊者（橙） | 既存構造を壊した外部参入者 | POSCO、Google/Meta、Sony |

**ANTAGONIST と DISRUPTOR の違い**:
- `ANTAGONIST` は事例の主軸時間軸の中で「拮抗して負けた」プレイヤー（当初は強かったが敗れる）
- `DISRUPTOR` は外部から来て既存構造を破壊したプレイヤー（POSCO は日本鉄鋼業の「勝利の方程式」を外から再演して相対化した）
- PROTAGONIST が後の事例でDISRUPTORに破壊される場合(日本鉄鋼がPOSCOに）、その DISRUPTOR を同じ事例に入れる

**注**: `ENABLER` / `REGULATOR` / `SUPPLIER` 等も DB に入れてよいが、比較ビューには表示されない（詳細は事例ページのサイドバーに出る）。

## 7. Source の扱い

- **一次情報 > 二次情報 > 三次情報**(`SourceKind.PRIMARY_DOC` を優先)
- **Published にする時までに、最低1ソースを紐付ける**
- **Source は DB で一元管理**。同じ本を複数事例で使う時は既存の Source を参照(重複作成しない)
- **`locator`** にはページ番号や章番号を書く。後から検証可能にする。
- **Wikipedia のみ**のソースは `status: draft` 止まりにする(published にしない)
- **draft 段階では推定情報や広く伝わる数字も許容**するが、published までに一次情報で裏取りする

## 8. 事実と解釈の分離

- **`Event.description` = 事実**: 誰が / いつ / 何を / どうした
- **本文 `## 転換点の意味` = 解釈**: なぜそれが転換点か
- **本文 `## 考察・私見` = 私見**: 自分がどう思ったか
- **`Lesson` = 蒸留**: 事実から抽出した構造

**検証**: `Event.description` を読んで「これは事実か?」と自問した時に、出典を当てられるなら事実。当てられないなら解釈なので本文に移す。

## 9. Status の運用

- `draft`: 書きかけ。frontmatter と俯瞰だけでも OK
- `review`: 本文が一通り埋まり、自分で数日寝かせて見直す段階
- `published`: 自分の中で完成。最低条件:
  - 俯瞰・構造分析・考察 が埋まっている
  - Event が3件以上、うち最低1件が turning point
  - Lesson が1件以上
  - Source が1件以上
  - 出典が一次情報に裏取りされているか、少なくとも定評のある二次情報である

## 10. slug の付け方

- **kebab-case**(英語小文字とハイフン)
- **短く**、3〜4語以内
- **永続**: 一度決めた slug は変えない(URL が壊れる)
- **例**: `tdl-founding`, `tokaido-shinkansen`, `containerization`, `subsea-cable-ownership`

## 11. Markdown と DB の境界(重要)

### Markdown に入れるもの
- frontmatter: `slug / title / one_liner / summary / status / themes / tags / period_start / period_end`
- 本文: `## 俯瞰` / `## 構造分析` / `## 転換点の意味` / `## 反対と執念` / `## 考察・私見`

### Markdown に入れないもの(DB 正本)
- `events` — admin UI の Event タブから
- `entities` — admin UI の Entity 紐付けから
- `lessons` — admin UI の Lesson タブから
- `sources` — admin UI の Source 紐付けから
- `case_relations` — admin UI の Relation タブから

### 下書き段階での例外
最初の下書きは `drafts/<slug>.yaml` に構造データの草案を置く。ここから admin UI へ手入力する。`drafts/` は一次的な置き場所で、DB 投入後に削除しても良いし、アーカイブとして残しても良い(運用判断)。

## 12. 本文の5セクション構造の意味

- **俯瞰**: 読者が最初の60秒で全体像を掴める要約。400〜800字。
- **構造分析**: 事実ベース。「何が起きていたか」「なぜそうなったか」「構造図」の3小節。本文の中心。
- **転換点の意味**: DB に登録した turning point について、「なぜ転換点か」の解釈。Event の description とは役割が違う(事実 vs 意味づけ)。
- **反対と執念**: 「当時どのような反対論が存在したか」と、「それを押し切った当事者たちの執念・退路の断ち方・人間関係」を記述する層。巨大プロジェクトの成否は制度や構造だけでは説明しきれず、反対論の分厚さと個人の執念の両方を書くことで、事例の「なぜ成立したか」が立体的になる。
- **考察・私見**: 私見・気づき・現代への転用。事実ではない層。読み返した時に「当時の自分の頭の中」が残る。

**注**: 残った疑問や調査メモは本文の最終出力には含めない。必要なら `drafts/<slug>.yaml` のコメントや別の調査ノートに退避する。

## 13. 他事例との関係性の扱い

- **CaseRelation は DB 管理**。本文には直接書かない。
- ただし本文 `## 考察・私見` の中で「X の事例と似ている」のように言及するのは構わない。
- **CaseRelation の relation_type** は現時点で6種類: `CONTRAST` / `PRECEDENT` / `DERIVATIVE` / `SIMILAR_MECHANISM` / `SAME_INDUSTRY` / `SAME_PATTERN`

## 14. パターン記事（Notes）の扱い

`content/notes/*.md` に置く、事例とは別種類のコンテンツ。UI 上は「パターン」として露出する（URL は `/notes/`）。**事例を横断して現れる構造的パターンを、独立した記事として扱う**場所。概念辞書や教科書ではない。

### Cases / Themes / Notes の役割分担

| | 役割 | 保存先 | 事例との関係 |
|---|---|---|---|
| **Case** | 個別事例の構造化された記録 | Markdown + DB | これが主役 |
| **Theme** | 事例を分類する軸（Case に紐付くタグ的役割） | DB のみ | 事例を束ねる |
| **Note** | 事例を横断して現れる構造的パターンの記事 | Markdown のみ | 事例を別角度で読ませる |

Theme が「分類」なら、Note は「読み方」。同じ概念が Theme と Note の両方に存在してもよい（例: `asset-operation-split` テーマ と `asset-operation-separation` パターン記事）。

### Note の最低要件

- **関連事例 3 件以上**を `related_cases` に持つ（単独で完結する "概念論" は書かない）
- 本文は基本 **8 節構成**:
  1. これは何か
  2. なぜ起こるのか / なぜ重要か
  3. どんな条件で強まるのか
  4. 典型的な進行パターン / 見取り図
  5. よくある誤解
  6. 既存事例で見るとどう見えるか
  7. 事業開発・投資判断への示唆
  8. 関連して読むべき既存事例
- **「これは何か」は厚めに書く**。このパターン自体がどのようなものか（何が起きるか、誰が登場するか、特徴的な動き、似て非なるものとの違い）を記述する節であり、「この記事が何を扱っているか」というメタ説明ではない。短く済ませないこと。
- 第 6 節では、**related_cases 各事例について「このパターンで読むとどう見えるか」** を1段落ずつ書く。単なる事例要約ではない。
- `published` にするのは、上記がすべて埋まり、他事例で読み返しても整合がとれる状態になってから。

### Note を書かない方がよいケース

- 関連事例が 1〜2 件しかない → まだ記事化しない。事例をもう少し増やしてから。
- 既存事例と結びつかない純粋な概念論 → そもそもこの DB の範疇外。別の場所に書く。
- Theme の説明（カテゴリの定義）と重複する → Theme 側の `description` で足りるかをまず検討。

### DB 連携

Note は Markdown のみで完結し、DB には入れない。関連付けは frontmatter の `related_cases` / `related_themes` で slug 参照する。参照先が DB に存在するかどうかは、ページ描画時にチェックする。

### 用語の方針

- 本文では「レンズ」という語を使わない。「型」「パターン」「構造」「視点」「枠組み」などで言い換える。
- 「解説」という表題も使わない。UI 上の見出し・本文中の呼称とも「パターン」で統一する。
