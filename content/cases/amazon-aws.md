# AmazonとAWSの成立——書店から始まった企業が「クラウド」というインフラ産業を創造した軌跡

## 俯瞰（Overview）

1994年7月、ジェフ・ベゾスは妻と共にニューヨークからシアトルへ車で横断しながら、後部座席でビジネスプランを書いた。目的地はガレージ。商品は書籍。武器は「インターネット利用が年間2,300%増加している」という統計から導いた確信——「物理的な棚に縛られる書店より、無限の在庫を持てるネット書店の方が構造的に有利だ」。

当初の投資家プレゼンでは「70%の確率で失敗する」と自分で言っていた。それでも「後悔最小化フレームワーク」——80歳になった自分が振り返ったとき、やらなかったことの後悔と、やって失敗した後悔のどちらが大きいかを問う思考実験——で参入を決めた。

Amazon.comが1995年7月に一般公開されると、最初の30日で全米50州と45カ国に書籍を届けた。1997年のIPO時、株式公開資料にベゾスは書いた——「これはDay 1だ」。

2000年のドットコムバブル崩壊でAmazonの株価は$106から$6に落ちた（▲94%）。社内では「Amazon.org（非営利）になる」という冗談が飛んだ。この生存危機が、逆に構造的な思考を生んだ。ベゾスは「なぜ自分たちが生き残れるか」を問い直し、「小売業者ではなく、テクノロジーカンパニーとして小売業をやっている」という再定義に至った。

2002年頃、ベゾスは社内命令を出した——全チームは自分のデータと機能をAPIとして公開し、その設計を「外部公開可能」にすること。違反者は解雇。この命令（後に「Bezos APIマンデート」と呼ばれる）は、Amazonの内部インフラを「製品」として設計することを強制した。2003年、アンディ・ジャシーがベゾスの影（シャドー＝技術顧問）として、「インフラをサービスとして外部に売る」構想を文書化した。

2006年3月、Amazon S3（Simple Storage Service）が公開された。同年8月、EC2（Elastic Compute Cloud）が公開された。使った容量だけ課金される、クレジットカード1枚で始められる、インフラ——これがAWS（Amazon Web Services）の誕生だ。

2015年、Amazonは初めてAWSの独立した財務数字を開示した。年間売上$78億、営業利益率30%超——当時のAmazon全社の営業利益の大部分をAWSが稼いでいた。「書店から始まった会社の子会社が、米国最大のクラウド事業者になっていた」という事実が資本市場に可視化され、Amazonの再評価が始まった。

## 構造分析

### 何が起きていたか

**書店から「あらゆるものの店」へ（1994〜2000年）**

Amazonは書籍から始めた理由がある——SKU（商品種類）が多く（300万冊以上）、物理書店の棚には収まらない、標準化された商品で返品リスクが低い、という理由だ。書籍での成功後、音楽（1998年）・DVD・家電・おもちゃへと品目を拡大した。「地球上で最も多くの商品を最も便利に買える場所」というビジョンは1997年時点で明言されていた。

成長のためにAmazonは膨大なIT投資を続けた。ピーク需要（クリスマスシーズン）に対応するサーバー・ストレージ・データベース・ネットワークを、年中稼働する通常時の5〜10倍規模で保有する必要があった。これが後に「余剰キャパシティを貸し出す」という発想につながったとされるが、実際はそれより深い洞察があった。

**ドットコムバブル崩壊と「再定義」（2000〜2003年）**

2000〜2001年、Amazonの株価は$106→$6に下落した。社債の格付けはジャンク（投機的）水準に落ち、倒産を予想するアナリストもいた。ベゾスは徹底的なコスト削減と同時に、「自分たちは何者か」という根本的な問いを社内に投げかけた。答えは「テクノロジーカンパニーだ——我々はテクノロジーを使って小売業をやっているのであり、小売業者ではない」。

2001〜2002年、ベゾスはAmazonの内部設計に問題があると診断した。各チームが独自の方法でシステムを組み、データを共有するには「直接接続」を使っていた。これはモノリシック（巨大な一枚岩）な設計で、スケールすると維持コストが爆発的に増加する。

**Bezos APIマンデートと内部のサービス化（2002〜2005年）**

ベゾスが出した命令は歴史的な内部メモとして伝わっている。その骨子は：
- 全チームはデータと機能をサービスインターフェース（API）経由で公開する
- チーム間の通信はAPIのみ（直接接続・共有データベースは禁止）
- 全APIは外部公開可能な設計にすること
- 例外なし、違反者は解雇

この命令が生んだ副産物は大きかった。「外部公開可能な設計」を強制されることで、Amazonのエンジニアは「自分たちが作っているインフラは製品だ」という思考様式に切り替わった。信頼性・ドキュメント・バージョン管理・SLA（サービスレベル保証）が内部インフラにも適用され始めた。

2003年、アンディ・ジャシー（ベゾスの"シャドー"として2年間働いていた）は、「Amazonが構築した信頼性の高い、スケーラブルなインフラを他社に提供するビジネス」の詳細な設計文書を作成した。これがAWSの事業計画書の原型だ。

**AWS公開とクラウドコンピューティングの誕生（2006〜2010年）**

2006年3月、Amazon S3（オブジェクトストレージ）が公開された。料金はGB当たり$0.15/月。同年8月、EC2（仮想コンピューティング）が公開された。スモールインスタンスは時間当たり$0.10。

この料金設定が持つ意味は革命的だった。従来、スタートアップがサーバーを持つためには数百万円の設備投資（物理サーバー購入、データセンターのラック代、SE要員費）が必要だった。AWSは月数百円から始められる変動費にした。Dropbox・Airbnb・Netflix等の次世代スタートアップがAWS上で生まれた。

NetflixのAWS移行（2008〜2016年）は象徴的な事例だ。自社データセンターの破壊的な障害（2008年）を機に、NetflixはすべてのITインフラをAWSに移行する決断をした。「世界最大規模のAWS顧客」として、Netflixはマルチリージョン展開・自動スケーリング・カオスモンキー（意図的に障害を起こして耐障害性を鍛えるツール）等の高度な活用法を開発し、それをオープンソースとして公開した。NetflixがAWSの「最先端利用のショーケース」として機能し、企業のAWS移行を加速させた。

**AWSがAmazonを逆転する（2010〜2015年）**

2013〜2014年頃から、内部でも外部のアナリストも「AWSがAmazon全体の利益の大部分を生んでいる」と推計し始めた。Amazonの小売部門は薄利（営業利益率3〜5%）でありながら、AWSは30%以上の営業利益率を持つと推定された。

2015年第1四半期、Amazonは初めてAWSの財務数字を独立して開示した。年間換算売上$78億、営業利益率26%（後に30%超）。この開示が株式市場のAmazon評価を根本的に変えた——Amazonは「利益の出ない小売業者」ではなく「高収益なクラウドビジネスを持つコングロマリット」として再定義された。

### なぜそうなったか（力学）

**「後悔最小化フレームワーク」と長期思考**

ベゾスの意思決定の特徴は「10年後を現在地として考える」長期思考だ。書籍を選んだのも、AWSを始めたのも、「今すぐ採算が取れるか」ではなく「長期的に価値があるか」という問いから来ている。IPO後も「四半期利益の最大化より長期価値の最大化」を株主に明言し、Amazonは20年近く営業利益をほぼゼロにしながら再投資し続けた。この長期思考が競合（Barnes & Noble、後にeBay・Walmart）との差別化の根本にある。

**フライホイール（自己強化する成長サイクル）**

ベゾスが2001年にナプキンに描いたとされる「フライホイール」図：
- 低価格 → 顧客増加 → 利用頻度増加 → セラー（マーケットプレイス出品者）増加 → 品揃え拡大 → 顧客体験向上 → 低価格（循環）
- 規模拡大 → コスト構造改善 → 低価格（別のサイクル）

このフライホイールはAWSにも適用される：
- 低価格 → 顧客増加 → 規模拡大 → インフラコスト低下 → さらなる低価格（循環）
- 多様な顧客の要求 → サービス拡充 → エコシステム拡大 → さらなる顧客増加（循環）

**APIマンデートの「設計の外部性」**

Bezos APIマンデートが生んだ副産物として、Amazonのエンジニアリングカルチャーが「製品思考」に変わった。内部ツールをAPIとして設計すると、「使いやすいか」「ドキュメントが揃っているか」「信頼性があるか」という製品品質の基準が自然に適用される。これが外部公開（AWS）への心理的・技術的障壁を下げた。「内部問題の解決が外部製品になる」という経路はAWSに限らず、Amazonの多くのサービス（フルフィルメントbyAmazon、物流サービス）に共通する。

**CapexのOpex変換とイノベーションの加速**

AWSが根本的に変えたのは「インフラを持つためのコスト構造」だ。サーバー購入・データセンター契約・SE要員の固定費（Capex）を、使量課金の変動費（Opex）に変換した。これにより：
- スタートアップが数億円の資金調達なしに事業を始められる
- 失敗コストが激減し、実験の回転速度が上がる
- 事業規模に比例してコストが動くため、スケールアップ時の追加調達が不要になる
この「イノベーションの摩擦を下げる」効果は、2006〜2015年のスタートアップ爆発（Airbnb・Uber・Spotify等が全員AWSで始めた）の主要因の一つだった。

**既存プレイヤーの「イノベーターのジレンマ」**

IBM・HP・Sun Microsystemsは自社のエンタープライズサーバー・ストレージビジネスで高い利益率（30〜50%）を享受していた。AWSは時間当たり$0.10という低価格でこれを代替し始めた。既存プレイヤーにとって、AWSに対応することは「自分の高収益ビジネスを自分でカニバル（自己侵食）すること」を意味した。IBMがクラウドへの本格移行に遅れたのは合理的な短期判断の結果だったが、長期的には市場を失った。これはクレイトン・クリステンセンが「イノベーターのジレンマ」で描いた構造と同型だ。

## 転換点の意味

### TP1: 2002年 Bezos APIマンデート

この転換点の意味は「Amazonが自分たちをインフラ企業として設計し始めた瞬間」だ。メモそのものは数百字の箇条書きだが、その含意は巨大だった——「自分たちの内部ツールを外部製品と同じ品質で作れ」。この命令なしに、Amazonの内部インフラは「外部に出せない複雑な固有システム」のまま留まっていたはずだ。APIマンデートは「製品として設計する」という思考の種を組織全体に植えた。アンディ・ジャシーがAWSの事業計画書を書けたのは、この思考様式が社内に存在していたからだ。

また、このマンデートは組織設計の観点からも重要だ。「チーム間の通信はAPIのみ」という規則は、チームを自律的なサービスとして分離することを強制した。これが後の「2ピザチーム（ピザ2枚で食べられる規模のチーム）」という組織設計原則とつながり、Amazonが数十万人規模になっても機敏さを保つ組織構造の基盤になった。

### TP2: 2006年 AWS公開（S3・EC2）

クラウドコンピューティングの「誕生の瞬間」として最も象徴的な転換点だ。Salesforce（1999年）やRackspace（2001年）など先行サービスはあったが、AWSが変えたのは「規模」と「標準化されたAPIへのアクセス」だ。どのエンジニアも同じAPIを使って同じインフラにアクセスできる——これが「クラウド」という産業カテゴリーを定義した。

AWS誕生時、IBMやHP・SunはAWSを「おもちゃ（toy）」と見ていた。「エンタープライズのミッションクリティカルな用途に使えるはずがない」という評価だった。この評価は2008〜2010年のNetflixのような先進的ユーザーによって覆された。「おもちゃ」からエンタープライズへの移行は、Amazonの意図的な設計（SLA保証・リージョン冗長・コンプライアンス認証）によって段階的に達成された。

### TP3: 2015年 AWSの独立財務開示

「透明化された事実が世界を変える」転換点だ。2015年以前、AWSの規模と収益性はアナリストの推計でしか分からなかった。開示後、二つのことが起きた。第一に、Amazonの株価評価が根本的に変わった——「利益の出ない小売業者」から「高収益クラウドビジネスを持つコングロマリット」への再定義だ。第二に、MicrosoftとGoogleがクラウドへの投資を急加速した——「AWSがこんなに儲かっているなら、市場は本物だ」という確認だった。Azureはこの開示後に投資が加速し、Google Cloud Platform（GCP）も本格化した。逆説的に、AWSの透明化が競合の本格参入を招き、クラウド市場全体のパイを拡大した。

## 反対と執念

2000年のドットコムバブル崩壊後のAmazonは「次に倒産する会社」と呼ばれた。株価$6の時点で、財務アドバイザーからは事業を大幅縮小・売却する選択肢を提示された。ベゾスは拒否した。代わりに「長期のフリーキャッシュフローを最大化する」という原則に戻り、短期利益よりインフラ投資を選んだ。2003年末にAmazonは初めて通年黒字を達成し、「倒産懸念」は消えた。

AWS公開時、社内には反対があった。「Amazonは小売会社だ。なぜITサービス会社になるのか」という声は経営幹部レベルにも存在した。ベゾスの回答は「これは小売に必要なインフラを既に持っているから、限界費用が低い。追加投資なしに新しい事業を作れる」という論理だった。

NetflixがすべてのインフラをAWSに移行すると決めた時（2008〜2016年）、社内には「全システムをAmazonという競合他社に依存させる」という批判があった。当時AmazonはNetflixと映像配信で競合していた（Amazon Video）。NetflixのCTOが「Amazonは顧客情報を競合事業に使わない」という判断をしたのは、AWSのサービス分離（AWSはAmazon小売・ビデオとは独立した事業部として運営）に信頼性を置いていたからだ。この判断は後に「クラウドインフラは中立的なインフラであり、提供者のビジネス判断で使われない」という産業規範の形成につながった。

## 考察・私見

AmazonとAWSの歴史が示す最も重要な教訓は「コアビジネスのために構築した能力が、次のビジネスの基盤になる」という命題だ。Amazonは書籍販売のためにサーバーを買い、注文処理のためにデータベースを構築し、ピーク対応のために冗長性を設計した。これらはすべて「小売のコスト」だったが、APIマンデートによって「製品として設計する」思考に変わった瞬間、コストが資産に転じた。

セブン-イレブンがPOSシステムを「コスト」から「競争優位の源泉」に変えたこと、TSMCが半導体製造という「コスト」を「独立したビジネス」として切り出したことと、同じ構造的転換だ。「自社の内部問題を解く能力が産業を変える」という命題はAmazonに特有ではなく、テクノロジー産業に繰り返し現れるパターンだ。

一方で、AWSの成功が引き起こした問題にも目を向けなければならない。クラウドコンピューティングは今や「インターネットの電力」に相当するインフラだが、AWS・Azure・GCPの3社が世界のクラウド市場を60〜70%支配する寡占構造を生んでいる。単一事業者への依存リスク（2017年のS3障害で多数のWebサービスが同時ダウン）、データ主権の問題（米国企業のサーバーに他国の重要データが保存される）、競争政策上の問題（AWSは競合スタートアップと同じ顧客の利用データを持つ）——これらは「インフラの民主化」と「インフラの新独占」という矛盾を示している。

ベゾスが2017年の株主書簡で書いた「Day 2は停滞であり、その後は凋落であり、最後は死である。だからDay 1のままでいる」という言葉は、自らが作ったものへの自己批判でもある。AWSがDay 1の精神を維持できるかどうかは、次の10年の問いだ。
